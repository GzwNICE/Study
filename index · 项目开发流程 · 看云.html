项目开发流程

小公司的开发流程（即我经历的）

小公司（尤其是初创公司）在产品开发上并没有太多规范约束，大体可以划分为五个大步骤。

  • 需求分析
  • 原型设计
  • UI/UE（效果图设计）
  • 产品开发
  • 测试上线

  1. 需求分析
    需求分析主要是对新产品进行功能分析，这块的主要参与人员即产品经理。如果是公司接的外包项目，在此需求分析之前产品经理（或领导）会与外包方先进行一个需求了解（我司是纯互联网公司，所以基本不做外包）；如果是公司内部的项目，可由产品经理直接进行需求分析，最终出一份初步的需求文档，文档主要内容即各个大功能和细化的小功能。
    随后一般产品经理会联系项目经理沟通资源调配，项目经理安排前端后端测试等人员，全体人员开一次项目启动会议，主要内容是项目经理介绍项目的出发点，项目的大体功能，在此期间其他人员也可以对项目中的功能提出可行性分析及必要性的意见（主要是项目经理主导），由此得出一份整改后的需求文档。

  2. 原型设计
    产品经理根据需求文档借助软件工具制作原型图，并在制作完成后交由UI涉及效果图或直接交由开发人员。

  3. UI/UE (效果图设计)
    一般对外项目讲究美观、用户体验，所以UI要根据原型图设计出漂亮布局的界面。大公司的话可能还会有UE（user experience）设计，主要针对交互做优化。
    而对于内部后台管理项目，此过程基本可以省略，即第二步所述直接将原型图交于开发者，开发者利用现有的组件库、样式库进行快速开发。

  4. 产品开发
    针对我司情况的一些表述，当然也能代表大多数公司。此处省略效果图切图环节，内部项目不需要效果图，靠开发发挥；外部项目切图部分有些交给UI做，我司是直接由前端负责

  • 过去
    开发根据自身的技术栈及公司的技术沉淀结合进行独立开发，通过公司自建的gitlab服务进行代码更新同步，后期配合测试fix bug。

弊端：公司不严格审查代码进度，仅仅有一个deadline，容易延期。

  • 现在
    4.1. 开发初期，项目经理会根据需求文档“禅道”平台划分大任务模块及子任务。开发人员在其基础上，对子任务进行进一步的细化及评估，约定开发时间周期；从而预估出本周及未来一周的工作计划。
    4.2. 开发人员按照约定的任务逐步开发，并及时在“禅道”平台更新完成进度
    4.3. 每周约定一天（一般为周一）对上周任务完成进度作总结并及时更新本周工作计划。
    4.4. 项目交付，测试接入，配合测试fix bug。
    （如果是前后端分离的项目，涉及到一个前后端联调的过程）

优点：按时交付，实时追踪进度。

  1. 测试上线
    上一步即提到，开发人员将系统推上测试环境，测试对系统进行体验测试，双方在“禅道”平台配合fix bug。经测试确认，即可初步上线接受第一批测试用户。

补充
上文开发流程中没有提到过的（或是未接触过的）

  1. code review
    代码审查机制，开发团队成员定期对代码进行二次审查。
  2. 自动化测试
    人工测试终究是慢的，高级测试一般都搞自动化，一次编写，多次运行。
  3. 技术选型
    我司对于技术、开发组件的沉淀还处于初期阶段，在项目开启时对技术的选型没有过多要求，即前端工具无限制，只要完成任务；后端就是一直在用的技术（但是没有过多的解决方案沉淀下来）。而一些发展比较规范的公司应该是技术和业务分离，有沉淀的诸多解决方案及组件库之类的东西。
  4. 接口设计
    我司在开发中是后端不断开发不断完善接口文档，前端预留接口，先mock数据，最后再对接。而我感觉在项目初期就可以根据需求文档进行沟通，先实现一个初步的接口文档。这样在后期不需要频繁的更改交互字段。
  5. DevOps
    这算是一个新型的项目开发运维模型，大公司应该采用的比较多。
